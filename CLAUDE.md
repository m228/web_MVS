# CLAUDE.md

Этот файл даёт указания Claude Code (claude.ai/code) при работе с кодом в этом репозитории.

## Что это

Веб-панель управления промышленными **GigE Vision камерами** (Hikrobot, напр. MV-CS050-10GC) и обычными **RTSP/IP-камерами** — самодостаточная замена настольной программе Hikrobot MVS. Бэкенд на FastAPI отдаёт MJPEG-видеопоток и фронтенд на чистом JS. Интерфейс и весь текст логов/комментариев — на русском.

GigE-камеры работают через `harvesters` → `genicam` → GenTL-продюсер (`Driver/MvProducerGEV.cti` + его runtime `Driver/MvCameraControl.dll`). RTSP-камеры идут через OpenCV/FFmpeg.

## Запуск и проверка

```bash
# Запуск приложения (Windows): активирует .venv, затем uvicorn на 0.0.0.0:8000

uvicorn app:app --host 0.0.0.0 --port 8000

# Диагностика GigE-драйвера / перечисления камер и падений create() (только чтение, ничего не меняет):
python diag.py

# Проверка синтаксиса без установки зависимостей камеры (работает на локальной машине разработки):
py -3.11 -m py_compile camera_core.py app.py net_tools.py
```

**Тестов нет**, линтер/форматтер **не настроены**. Целевой Python — 3.11; зависимости в `requirements.txt` (внимание: файл в кодировке UTF-16).

## Важно: топология разработки и развёртывания

**Камеру нельзя проверить локально.** Код пишется на локальной машине, но реальная GigE-камера подключена к ОТДЕЛЬНОЙ машине «Server» (`C:\Users\Server\...\web_MVS`, со своим venv на Python 3.11). На локальной машине обычно **нет** `harvesters`/`genicam`/`numpy`/`opencv`.

Цикл проверки: правка кода → пользователь коммитит/пушит → тянет на Server → запускает там `diag.py` или приложение → присылает вывод/логи. Поэтому:
- Не предлагай «запустить приложение здесь» для проверки поведения камеры/драйвера.
- Для отладки драйвера пиши самодиагностику, которую пользователь прогонит на Server, и опирайся на панель **«История событий»** в приложении (на базе `logger.py`).
- Свои правки проверяй локально через `py_compile` (зависимости не нужны).

Часть сетевых функций (`net_tools.py`: jumbo-кадры, фильтр-драйвер GigE) требует запуска приложения **от имени администратора**; чтение `status()` — не требует.

## Архитектура

**Бэкенд** тонкий: `app.py` — плоский список эндпоинтов `@app.get`, которые делегируют единому глобальному `manager` (экземпляр `CameraManager` из `camera_core.py`). Почти всё идёт через `manager`.

`camera_core.py` (ядро, ~1800 строк):
- `CameraManager` — владеет `Harvester`, один раз загружает драйвер `.cti`, сканирует сеть и является **реестром** воркеров (`get(serial)` → `CameraWorker`, `get_rtsp(serial, url)` → `RtspCameraWorker`). Один глобальный экземпляр: `manager`. Создание воркеров сериализовано через `_registry_lock`.
- `BaseCameraWorker` — общая логика автосохранения фото/видео и метрик для обоих типов камер.
- `CameraWorker` (GigE) — открывает GenICam nodemap, применяет настройки, а `generate()` отдаёт MJPEG-поток (multipart) через `ia.fetch()`. Операции по control-каналу (чтение/смена IP, инфо) сериализованы на камеру через `_control_lock` и кратко кэшируются, потому что фронтенд шлёт много параллельных запросов, а единственный control-канал отдаёт AccessDenied (-1005) при конкуренции.
- `RtspCameraWorker` — тот же интерфейс стрима/сохранения поверх `cv2.VideoCapture` (принудительно по TCP).

**Модель стриминга:** каждый видеопоток — это `StreamingResponse(worker.generate(...), media_type="multipart/x-mixed-replace; boundary=frame")`. Остановка кооперативная: `close()` ставит `running=False` (разбор происходит в `finally` генератора); `force_close()` немедленно останавливает/уничтожает acquirer. Фронтенд закрывает потоки при выгрузке страницы, чтобы не оставлять «висящие» acquirer'ы.

**Идентификация устройства нетривиальна.** Hikrobot показывает *одну и ту же* камеру как несколько записей (по одной на сетевой интерфейс), с общим серийником и даже общим GenTL `id_`. Поэтому камера ключуется по **serial_number**, но конкретная запись выбирается по `device_handle` (`id_#index`, приоритет 1), затем по `interface_id` (родительский интерфейс, приоритет 2). `create_acquirer()` перебирает записи в порядке: предпочтительная → доступная → запасная. Эндпоинты принимают необязательные `interface_id`/`device_handle`, чтобы зафиксировать запись.

`logger.py` — общий на процесс кольцевой буфер в памяти (`deque(maxlen=1000)`) структурированных событий с инкрементными ID. Фронтенд опрашивает `/api/debug/logs?since_id=N`. Это основной канал наблюдаемости (см. раздел про топологию). Используй `log_event(source, message, level, payload)`.

`net_tools.py` — обёртка над PowerShell (`Get/Set-NetAdapterAdvancedProperty`, `*-NetAdapterBinding`) для переключения jumbo-кадров и фильтр-драйвера GigE. Работает через локаль-независимый `RegistryKeyword '*JumboPacket'` и форсирует вывод в UTF-8.

`rtsp_store.py` — крошечное хранилище в JSON (`rtsp_cameras.json`) сохранённых RTSP-камер, ключ — URL.

**Фронтенд** (`page/`): пять HTML-страниц, отдаваемых сервером (`index`, `camera`, `rtsp`, `multi`, `network`), загружаются по своим маршрутам в `app.py`. JS в `page/static/js/` — простые глобалы (без бандлера/модулей): `api.js` выставляет `window.CameraApi` / `RtspApi` / `NetApi`; `logger.js` выставляет `window.AppLog`; логика страниц в `camera.js`, `rtsp.js`, `multi.js`, `network.js`, `index.js`; общие `theme.js`, `ui.js`. Скрипты подключаются тегами `<script src>` внизу каждого HTML-файла.

## Совместимость GenTL / Hikrobot (прочитай перед правкой кода драйвера)

`camera_core.py` содержит костыли совместимости для известных причуд Hikrobot+genicam. Не удаляй их, не разобравшись в `bug.txt`:
- `_patch_genicam_register_event()` (выполняется при импорте): продюсер Hikrobot не реализует `register_event` для модулей System/Interface/Device; genicam 1.5.1 бросает `UnicodeDecodeError` (из не-UTF-8 байтов продюсера) вместо `NotImplementedException`, который harvesters не ловит → `create()` падает (плавающе). Патч превращает это обратно, чтобы harvesters его проглотил. Получение кадров (`DataStream`) не трогается.
- `_explain_error()` / `_decode_gentl_message()` — декодируют не-UTF-8 сообщения GenTL (latin-1) и сопоставляют коды GenTL через `GENTL_HINTS` (напр. -1003 не поддерживается, -1005 занято/AccessDenied, -1011 таймаут кадра/потеря пакетов).
- `_discover_cti()` / `_find_mvs_runtime()` — драйвер лежит в `Driver/`; бандл используется только если его runtime-DLL находится рядом с `.cti`, иначе откат на системный MVS (`GENICAM_GENTL64_PATH`). Переопределяется через `MVS_CTI_PATH`.

`bug.txt` — текущий журнал багов камеры и их фиксов; сверяйся с ним по симптомам драйвера (напр. падение register_event, «раньше работало»). `task.txt` — чек-лист фич.

## Соглашения

- Серийник `"DA123123"` — захардкоженные **mock**-данные (фейковая камера/IP `192.168.2.10`) в нескольких местах (`get_ip`, `get_info`, `list_devices_grouped`, `scan_cams`) для проверки UI без железа. Помечены `пробные данные(потом убрать)` — не трогай без отдельной просьбы.

- Сохранённые медиа идут в `dataset/` (фото) и `Videos/` (MJPG `.avi`), каталоги создаются по необходимости.
- Эндпоинты только GET и логируют `log_event` на входе и на результате; сохраняй этот паттерн при добавлении эндпоинтов.
