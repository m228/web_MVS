# Упаковка и обновление web_MVS на нескольких машинах

Цель: держать несколько копий приложения на «чистых» Windows-машинах (без Python) и
легко обновлять. Приложение собирается в самодостаточный бандл (PyInstaller) и
раздаётся одним **архивом** через GitHub Releases.

## Что попадает в архив, а что нет

Архив `web_MVS_v<версия>.zip` самодостаточен — этого достаточно для работы:

```
web_MVS.exe        ┐ из архива (перезаписываются обновлением)
_internal\        ─┤ Python, зависимости, page\, Driver\ (.cti+DLL), VERSION
run.bat           ┘ запуск от администратора
```

НЕ входят в архив (и поэтому не затираются обновлением):

```
dataset\           <- фото
Videos\            <- видео
rtsp_cameras.json  <- сохранённые RTSP-камеры
update.bat         ┐ «загрузчик» — он САМ качает архив, поэтому лежит снаружи архива
update.ps1         ┘ (берётся из репозитория, кладётся рядом один раз)
```

Все эти файлы лежат в **одной папке** рядом с `web_MVS.exe` (данные пишутся туда же,
куда распакован архив).

## А. Сборка релиза (на машине с Python 3.11 и зависимостями)

Один раз подготовить окружение:

```bat
py -3.11 -m venv .venv
.venv\Scripts\pip install -r requirements.txt
```

При каждом релизе:

1. Поднять номер в файле `VERSION` (напр. `1.0.1`).
2. Запустить `build.bat` — соберёт `dist\web_MVS\` и упакует `dist\web_MVS_v<версия>.zip`
   (с `run.bat` внутри).
3. Опубликовать релиз (нужен авторизованный `gh` — GitHub CLI):

   ```bat
   gh release create v1.0.1 dist\web_MVS_v1.0.1.zip -t v1.0.1 --generate-notes
   ```

> Тег релиза `vX.Y.Z` = содержимому `VERSION`; имя ассета должно начинаться с
> `web_MVS_` (по этой маске его ищет `update.ps1`).

## Б. Установка на целевой машине — два способа

**Способ 1 — вручную (просто и достаточно):**
1. Скачать `web_MVS_v<версия>.zip` со страницы Releases.
2. Распаковать в любую папку, напр. `C:\web_MVS\`.
3. Запустить `run.bat` (сам запросит права администратора) →
   откроется `http://localhost:8000`.

   Без админ-прав можно правым кликом по `web_MVS.exe` → «Запуск от имени
   администратора» (админ нужен для jumbo/фильтра GigE на странице «Сеть камер»).

**Способ 2 — через загрузчик (одна кнопка, и для установки, и для обновления):**
1. Создать папку `C:\web_MVS\`, положить туда `update.bat` и `update.ps1` из репозитория.
2. Запустить `update.bat` — скачает последний релиз и распакует рядом.
3. Запустить `run.bat`.

`update.ps1` использует `gh`, если он есть; иначе публичный GitHub API
(репозиторий должен быть публичным).

## В. Обновление

- **Из приложения (проще всего):** на главной странице — карточка «Обновление».
  Кнопка «Проверить обновления» сравнит версию с последним релизом; если есть новее —
  «Скачать обновление», затем «Обновить и перезапустить». Приложение само закроется,
  заменит файлы и поднимется заново (страница перезагрузится). Данные сохраняются.
- **Вручную:** скачать новый zip, распаковать **поверх** старой папки (с заменой).
  Данные (`dataset\`, `Videos\`, `rtsp_cameras.json`) не в архиве → сохраняются.
- **Через загрузчик:** запустить `update.bat` — остановит `web_MVS`, скачает релиз,
  обновит файлы приложения, данные не тронет. После — снова `run.bat`.

> Самообновление и `update.bat` требуют **публичного** репозитория (скачивание без
> авторизации).

Проверить версию: открыть `http://localhost:8000/api/debug/info` — там `version`,
`data_dir` (куда пишутся данные), версии genicam/harvesters и дата `.cti`.

---

## Почему так (а не иначе)

- **git + venv на каждой машине** — отпадает: целевые машины без Python, тяжёлый
  `pip install` (numpy/opencv/harvesters) долгий и хрупкий.
- **Один .exe (one-file)** — отпадает: распаковка во временную папку при каждом
  старте (медленно), нативная загрузка `.cti`/`MvCameraControl.dll` хрупкая.
- **Docker** — отпадает: GigE Vision требует raw-доступ к сети/NIC, jumbo-кадры и
  админ-PowerShell — в контейнере на Windows это боль.

## Если GitHub недоступен на целевой машине

Перенести `web_MVS_v<версия>.zip` вручную (флешка/сетевая папка), распаковать
содержимое в папку приложения (с заменой) и запустить `run.bat`.
